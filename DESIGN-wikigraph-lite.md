## Problem Statement
Сделать reader-first knowledge explorer (а не полный заменитель Википедии): более удобный и интуитивный интерфейс для обычного пользователя, где фокус на чтении и исследовании связей между темами.

## What Makes This Cool
Ключевой "wow"-фактор — режим исследования как карта знаний: пользователь не просто читает статью, а живо перемещается по смысловым связям, не теряя контекст.

## Constraints
- Проект в режиме "for fun" с фокусом на кайф и скорость результата.
- Нужен быстрый MVP, который можно показать и сразу получить реакцию.
- Нельзя начинать с тяжелого цикла создания собственного контента с нуля.
- Нужно вырезать лишнюю сложность классической вики для обычного читателя.

## Non-Goals (MVP)
1. Не делаем wiki-редактор и вики-разметку.
2. Не делаем аккаунты, роли и модерацию.
3. Не делаем мультиязычность.
4. Не делаем AI-ассистента в MVP.
5. Не делаем продвинутую персонализацию.

## Premises
1. Главное значение MVP — чтение + навигация по связям, а не полный административно-редакторский функционал Википедии.
2. Самый быстрый путь — новый UX-слой поверх существующего открытого контента/датасета.
3. Первая целевая аудитория — исследователи/энтузиасты, которым важна глубина переходов между понятиями.
4. Успех MVP измеряется тем, что пользователь делает 2-3 связных перехода и не теряет контекст.

## Data Contract
- Source: открытый набор статей и связей (Wikipedia/Wikidata API или ограниченный dump).
- Scope MVP: 20 статей в одном тематическом кластере (ручной эталон).
- Language: ru (MVP), без мультиязычности.
- Refresh cadence: ручное обновление набора для MVP, автообновление после MVP.
- Data guards: фильтрация битых ссылок, защита от циклических переходов, fallback к curated edges.
- Source of truth (MVP): curated edges; auto-edges только как suggested hints без влияния на обязательный маршрут.

## NFR (MVP)
- p95 загрузка страницы статьи < 2.5s.
- Отрисовка графа до 50 узлов < 500ms на целевом десктопе.
- Успешный backtrack по маршруту не менее 95%.
- UI должен оставаться usable при отсутствии части внешних связей (деградация без падения).

## Approaches Considered
### Approach A: Graph Reader Overlay (MVP)
- Summary: использовать существующий открытый контент (Wikipedia/Wikidata/API/dump), поверх него сделать новый интерфейс: быстрый ридер, граф связей, история маршрута.
- Effort: M
- Risk: Low-Med
- Pros:
  - Самый короткий путь к демонстрабельной версии.
  - Быстро проверяется ключевая UX-гипотеза.
  - Минимальный инфраструктурный риск на старте.
- Cons:
  - Зависимость от внешней структуры данных.
  - Ограниченный контроль над качеством исходных статей.
- Reuses: открытые датасеты, готовые графовые сущности, существующие индексы.

### Approach B: Curated Micro-Encyclopedia
- Summary: взять узкий домен и построить эталонный UX с тщательно отобранными материалами и связями.
- Effort: M-L
- Risk: Med
- Pros:
  - Максимальный контроль качества UX и структуры.
  - Четкий фокус на сценарии исследования.
- Cons:
  - Медленнее масштабируется.
  - Требует ручного контент-процесса.
- Reuses: headless CMS, кастомная онтология, редакторский workflow.

### Approach C: AI-Guided Exploration Layer
- Summary: добавить поверх граф-ридера AI-проводник (краткие объяснения, рекомендованные переходы).
- Effort: L
- Risk: Med-High
- Pros:
  - Сильный визуальный и продуктовый "wow".
  - Потенциально более высокое удержание.
- Cons:
  - Сложнее обеспечить достоверность.
  - Выше стоимость и сложность эксплуатации.
- Reuses: LLM + retrieval + safety/grounding.

## Recommended Approach
Выбрать Approach A (Graph Reader Overlay). Это дает самый быстрый и чистый путь проверить основную ценность идеи: действительно ли людям нравится исследовать знания через граф навигации в более дружелюбном интерфейсе.

## Technical Plan (2 недели)
- Week 1:
  1. Подготовить curated набор из 20 статей и adjacency-таблицу обязательных связей.
  2. Реализовать базовый ридер + sidebar граф + breadcrumb/история переходов.
  3. Выбрать storage MVP: SQLite + precomputed adjacency.
- Week 2:
  1. Дополировать UX маршрута (backtrack, breadcrumbs, ограничение узлов) и стабилизировать сценарии.
  2. Провести сценарные тесты на 5-10 пользователях.
  3. Подготовить backlog для V1 ingest и quality-threshold auto-edges.

## Risk Register
1. Низкое качество auto-edges -> fallback на curated edges.
2. API rate limits -> кэш + локальный snapshot.
3. Перегруженный граф -> ограничение узлов и приоритизация top-N связей.

## Open Questions
1. Нужен ли логин в MVP или достаточно анонимного ридера?
2. Какой формат экспорта пользовательского маршрута добавить в V1 (share-link или markdown summary)?

## Success Criteria
- Task success rate: >= 80% пользователей выполняют сценарий "найти 3 связанные темы".
- Median time-to-3-hops: <= 90 секунд.
- Backtrack success: >= 95% переходов назад без потери контекста.
- Route determinism: >= 95% (одинаковый старт+цель на одном snapshot дает одинаковый топ-3 маршрут).
- Usability score: средняя субъективная оценка >= 4.2/5 по вопросу "насколько удобно исследовать тему".

## Distribution Plan
- Формат доставки MVP: web-приложение с публичной ссылкой.
- Хостинг: простой web deploy (Vercel/Netlify/Cloudflare Pages).
- CI/CD: деплой по push в основную ветку после базовых проверок (build + smoke).
- Контентный источник: открытые API/дампы Wikipedia/Wikidata + периодическое обновление индекса.

## Roadmap
- MVP: статья + 1 тип связей (related topic) + история переходов + curated dataset 20x(5-10) + UI-ограничение max 7 узлов на экране.
- V1: авто-ingest + quality-threshold auto-edges + базовая фильтрация связей.
- V2: расширенные типы связей и персонализация траекторий.

Собери список из 20 статей одного тематического кластера (например, "машинное обучение" или "история Рима") и вручную определи для каждой 5-10 "обязательных" связей. Это станет эталонным тестовым набором для проверки качества граф-навигации в MVP.

## Design specification (`/plan-design-review`, без CLI-мокапов)

CLI `gstack design` для PNG-вариантов недоступен без `OPENAI_API_KEY` / `design setup`. Ниже — зафиксированные решения, чтобы вёрстка не превратилась в «AI slop».

### Information architecture
- Одна основная поверхность: **Article + Graph + Wayfinding** на одном экране (desktop).
- Навигация отвечает на trunk test: логотип/имя продукта слева сверху, явный заголовок статьи, крошки, где я в теме.

### Layout (desktop ≥1024px)

```
+------------------------------------------------------------------+
| [WikiGraph Lite]   Home > Кластер > Статья          [← Назад]     |
+----------+-----------------------------------+-------------------+
|          |  H1 Заголовок статьи            |  Связи              |
|  (опц.)  |  Подзаголовок / выдержка        |  [мини-карта графа] |
|  оглавл. |  --------------------------------|   o—o—o             |
|  статьи  |  Основной текст…                |   |                 |
|  якоря   |  …                              |   o + «ещё N»       |
|          |                                 |  подпись: curated |
+----------+-----------------------------------+-------------------+
```

- **Main:** 65–75% ширины. **Graph sidebar:** 25–35%, min-width ~280px.
- **Graph:** не более **7** узлов одновременно; остальное — «Показать ещё» (раскрытие списком, без раздувания канваса).

### Typography (конкретика)
- **Заголовки статьи:** `Literata` или `Source Serif 4` (Google Fonts), H1 28–32px, H2 22px, межстрочный 1.45–1.55 для body.
- **UI chrome / граф / крошки:** системный стек `system-ui, "Segoe UI", sans-serif` или `Inter` если подключаете один веб-шрифт.
- **Максимальная ширина меры:** 65–72ch для текста статьи.

### Цвет и материал
- Фон: тёплый off-white `#F7F5F2` или нейтральный `#FAFAF8`.
- Текст: `#1a1a1a` (не чистый `#000`).
- Акцент **один:** глубокий синий `#1e3a5f` или teal `#0f766e` — только ссылки, активный узел, primary button.
- Граф: узлы нейтральные `#e8e4de`, активный узел — акцент + 2px обводка; рёбра `#c4bdb5` 1px.

### Состояния (обязательны в MVP)
1. **Загрузка статьи:** skeleton в main, sidebar «загружаем связи…».
2. **Пустой граф (нет curated рёбер):** иллюстрация или иконка + «Для этой статьи пока нет связей в наборе» + ссылка «к списку кластера» (если будет).
3. **404 slug:** короткий текст + кнопка «Назад» + ссылка на старт кластера.
4. **Длинный заголовок (47+ символов):** перенос по словам; H1 не обрезать многоточием без tooltip.
5. **Ошибка битого JSON снапшота:** дружелюбный баннер «данные повреждены» + retry (перезагрузка).

### Взаимодействие
- Клик по узлу графа = переход; узлы визуально **кнопки** (радиус 8px, hover, focus ring).
- **Клавиатура:** Tab порядок: skip link → back → первый узел графа → основные ссылки в статье; Escape в модалях (если появятся).
- **Touch:** цели ≥44px для узлов на планшете.

### Mobile (<768px)
- Статья full width; граф — **нижняя шторка** или таб «Статья | Связи»; крошки сворачиваются в одну строку с горизонтальным скроллом.

### Доверие
- Футер или мелкая строка под статьёй: «Источник: снапшот Википедии (CC BY-SA), дата сборки» — без юридического шума, но видимо.

### Что сознательно не делаем (дизайн)
- Нет hero-баннеров, нет трёх колонок «feature cards», нет градиентных фиолетовых фонов.
- Нет иллюстраций stock «people pointing at laptop».
